## The Foxglove Process Check

### The Requirement

Before starting the robotic control script, it's essential to verify that no Foxglove processes are currently running. This might seem like a simple housekeeping task, but there are important technical reasons why this check is necessary.

### What is Foxglove?

Foxglove Studio is a modern robotics visualization and debugging platform. It's essentially the next-generation successor to tools like RViz (ROS Visualization). Foxglove connects to ROS2 systems and provides:

- **Real-time 3D visualization** of robot models and sensor data
- **Topic monitoring** to see messages flowing through the ROS2 graph
- **Data plotting** for analyzing sensor values, joint positions, and control signals
- **Log playback** for reviewing recorded robot sessions
- **Custom panels** for building tailored debugging interfaces

Think of it as the "developer tools" for robotics—similar to how web developers use browser DevTools to inspect network traffic and debug JavaScript.

### Why Foxglove Processes Must Be Stopped

There are several technical reasons why existing Foxglove processes can interfere with new control scripts:

#### 1. Subscription Competition

When Foxglove connects to a ROS2 system, it creates subscribers for topics it's visualizing. If you then start a control script that also subscribes to the same topics, you now have multiple subscribers competing for messages.

While DDS (the underlying communication middleware) can handle multiple subscribers, there are scenarios where this causes issues:

- **QoS Incompatibility**: If Foxglove and your script use different Quality of Service settings, one or both might fail to receive messages
- **Resource Contention**: On resource-constrained systems, multiple subscribers processing large message streams (like camera images) can cause CPU or bandwidth saturation
- **Message Queue Overflows**: If Foxglove's callback processing is slow, its message queues might fill up, potentially affecting overall system performance

#### 2. Port Conflicts

Foxglove can run in different modes:

- **WebSocket Bridge Mode**: Opens a port (typically 8765) to bridge ROS2 to web browsers
- **Native Mode**: Connects directly to ROS2 topics

If your control script tries to launch its own Foxglove bridge or uses the same port for a different purpose, you'll get a "port already in use" error. The script will fail to start, but the error might not be immediately obvious.

#### 3. Discovery Server Load

In the Atlas system, we learned that the discovery server at `192.168.1.123:11811` is critical for allowing distributed nodes to find each other. Foxglove registers itself with this discovery server.

If an old Foxglove instance is still registered but not functioning properly (perhaps because it lost network connection or the user closed the window without proper cleanup), it creates a "zombie subscriber" that:

- Clutters the discovery server's registry
- May still consume messages even though no one is viewing them
- Slows down discovery for new nodes
- Can cause confusion when debugging (you see subscribers that shouldn't exist)

#### 4. Configuration State Persistence

Foxglove can modify ROS2 parameters or maintain state about the system. If it's left running across different script launches, it might:

- Hold onto outdated parameter values
- Have cached topic configurations that don't match the new script
- Display stale data that doesn't reflect the current system state

This can lead developers astray during debugging, as they're looking at information that doesn't represent the actual running system.

#### 5. File Lock Conflicts

If Foxglove is recording data to a bag file (ROS2's data logging format), it holds a file lock on that bag. If your control script tries to:

- Append to the same bag file
- Read from the bag while Foxglove is writing to it
- Delete or move the bag file

You'll encounter file system errors or data corruption.

### How to Implement the Check

There are several approaches to ensuring no Foxglove processes are running:

#### Approach 1: Manual Check Before Launch

Add a check to your launch script that verifies no Foxglove processes exist:

```bash
#!/bin/bash

# Check for running Foxglove processes
FOXGLOVE_PIDS=$(pgrep -f foxglove)

if [ -n "$FOXGLOVE_PIDS" ]; then
    echo "ERROR: Foxglove processes detected:"
    ps -p $FOXGLOVE_PIDS -o pid,cmd
    echo ""
    echo "Please close Foxglove before starting the control script."
    echo "You can kill them with: killall foxglove"
    exit 1
fi

echo "✓ No Foxglove processes detected. Proceeding with launch..."

# Launch your control script
python control_script.py
```

This approach is non-invasive—it informs the user but doesn't automatically kill processes, preventing accidental data loss if Foxglove was recording something important.

#### Approach 2: Automatic Cleanup

For automated systems or when you're confident no important recording is happening:

```bash
#!/bin/bash

# Kill any existing Foxglove processes
echo "Checking for Foxglove processes..."
pkill -9 foxglove 2>/dev/null

if [ $? -eq 0 ]; then
    echo "⚠️  Killed existing Foxglove processes"
    sleep 1  # Give the OS time to clean up
fi

echo "✓ System ready. Launching control script..."
python control_script.py
```

The `-9` flag sends SIGKILL, which forcefully terminates the process. This is aggressive but ensures the process is truly gone. The `2>/dev/null` suppresses error messages if no Foxglove processes exist.

#### Approach 3: Python Integration

If your control script is in Python, you can perform the check within the script itself:

```python
import psutil
import sys
import time

def check_foxglove_processes():
    """Check for running Foxglove processes and warn user."""
    foxglove_processes = []

    for proc in psutil.process_iter(['pid', 'name', 'cmdline']):
        try:
            # Check if 'foxglove' appears in process name or command line
            if 'foxglove' in proc.info['name'].lower():
                foxglove_processes.append(proc)
            elif proc.info['cmdline'] and any('foxglove' in cmd.lower()
                                               for cmd in proc.info['cmdline']):
                foxglove_processes.append(proc)
        except (psutil.NoSuchProcess, psutil.AccessDenied):
            continue

    if foxglove_processes:
        print("ERROR: Foxglove processes detected:")
        for proc in foxglove_processes:
            print(f"  PID {proc.pid}: {' '.join(proc.info['cmdline'])}")
        print("\nPlease close Foxglove before starting this script.")
        print("Or run with --force-kill to automatically terminate them.")
        return False

    return True

def kill_foxglove_processes():
    """Forcefully terminate all Foxglove processes."""
    killed = False
    for proc in psutil.process_iter(['pid', 'name']):
        try:
            if 'foxglove' in proc.info['name'].lower():
                proc.kill()
                killed = True
                print(f"Killed Foxglove process (PID {proc.pid})")
        except (psutil.NoSuchProcess, psutil.AccessDenied):
            continue

    if killed:
        time.sleep(1)  # Wait for cleanup

    return killed

if __name__ == "__main__":
    import argparse

    parser = argparse.ArgumentParser()
    parser.add_argument('--force-kill', action='store_true',
                       help='Automatically kill Foxglove processes')
    args = parser.parse_args()

    if args.force_kill:
        kill_foxglove_processes()
    elif not check_foxglove_processes():
        sys.exit(1)

    print("✓ No Foxglove processes detected. Starting control system...")

    # Your control script code here
    # ...
```

This approach provides flexibility—users can choose whether to manually close Foxglove or have the script kill it automatically.

#### Approach 4: Systemd Service Management

For production deployments where the control script runs as a systemd service, you can declare Foxglove as a conflicting service:

```ini
[Unit]
Description=Robot Control Service
After=network.target
Conflicts=foxglove.service

[Service]
Type=simple
User=jasper
WorkingDirectory=/home/jasper/workspace/atlas
ExecStartPre=/usr/bin/pkill -9 foxglove
ExecStart=/usr/bin/python3 control_script.py
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

The `Conflicts=foxglove.service` directive ensures systemd won't allow both services to run simultaneously. The `ExecStartPre` command kills any stray Foxglove processes before starting.

### When Foxglove Should Run

So if Foxglove shouldn't run with the control script, when should it be used?

**Development and Debugging:**
- Running Foxglove alongside test scripts to visualize behavior
- Recording data during teleoperation sessions
- Analyzing logged data after experiments
- Monitoring system health during initial setup

**The Key Difference:**
When you're actively developing and want visualization, you start Foxglove **first**, then run your scripts with awareness that Foxglove is connected. You coordinate the startup.

When you're running production control scripts or automated tests, Foxglove should be **closed** to avoid interference and ensure deterministic behavior.

### Best Practices

1. **Document the requirement**: Add a note in your README that Foxglove should be closed before running control scripts.

2. **Provide helpful errors**: If your script detects Foxglove, show exactly which processes exist and how to kill them.

3. **Use consistent naming**: If you launch Foxglove with custom node names, document them so your check scripts can find them.

4. **Consider dedicated machines**: For production systems, run control scripts on a dedicated computer that never runs visualization tools.

5. **Logging instead of live visualization**: Instead of using Foxglove during control, record to bag files and visualize afterward. This eliminates interference entirely.

### The Broader Lesson: Clean State for Critical Systems

This Foxglove check exemplifies a broader principle in systems engineering: **critical operations should start from a known, clean state**.

In robotics:
- Clear stale ROS2 nodes before launching new ones
- Reset hardware controllers to default states
- Flush network buffers
- Clear temporary files from previous runs

In software development:
- Clear test databases between test runs
- Reset mock objects between tests
- Clear browser cache when debugging web applications
- Restart services after configuration changes

In DevOps:
- Use immutable infrastructure (fresh deployments rather than updates)
- Clear container volumes between runs
- Reset Kubernetes pods rather than hot-patching them

The small act of checking for Foxglove processes is part of this larger philosophy: ensure your starting conditions are predictable, eliminating sources of non-deterministic behavior.
